# Meal Planner Skill (一周食谱规划)

本 skill 根据用户的**饮食偏好**和**当周饮食期望**，从一个**已建立的菜品库**中挑选菜品，生成定制化的一周食谱。

## 何时使用本 skill（重要）

**只在用户明确点名调用时才执行本 skill 的工作流。** 触发示例：

- "用 meal-planner-skill 帮我排食谱"
- "调用一下食谱规划 skill"
- "run meal-planner-skill"
- 用户在对话中明确提到 `meal-planner-skill` 这个名字

**不要**因为以下情形自动启动本 skill：

- 用户随口问"今晚吃什么"、"鸡胸肉怎么做"
- 用户聊到减肥、营养、菜谱话题但**没**点名本 skill
- 用户分享食谱链接、问烹饪技巧、查热量

如果不确定用户是否要调用本 skill，**先用一句话问清楚**："你是想让我用 meal-planner-skill 给你排一份正式的一周食谱（会读取/更新你的菜品库和档案），还是只是聊聊吃的？"

> 关键原则：**所有菜品必须来自 `dish-library.md`，不得凭空猜想**。如果库中缺少能满足当周期望的菜品，必须先询问用户并把新菜加入库中，再规划。

## 文件结构

skill 文件夹下会出现以下文件（未存在的会按需创建）：

```
meal-planner-skill/
├── SKILL.md                       # 本文件（指令）
├── templates/
│   ├── user-profile.template.md   # 用户资料模板
│   ├── dish-library.template.md   # 菜品库模板
│   └── meal-plan.template.md      # 食谱模板
├── user-profile.md                # 用户资料（首次使用时创建）
├── dish-library.md                # 菜品库（首次使用时创建，会随使用迭代）
└── meal-plan-YYYY-MM.md           # 按月份分文件，每月一份；月内按周追加
                                   # 跨月的周以"该周周一所在月份"为准
```

## 主工作流（每次被调用都按此执行）

复制以下检查清单并跟踪进度：

```
任务进度：
- [ ] Step 1: 判断是否首次使用
- [ ] Step 2: 首次 → 引导建档；非首次 → 加载档案
- [ ] Step 3: 解析本次"周饮食期望"
- [ ] Step 4: 检查菜品库是否够用，必要时补充
- [ ] Step 5: 生成一周食谱并写入独立文件
- [ ] Step 6: 输出概要并询问是否需要迭代
```

### Step 1: 判断是否首次使用

- 检查 skill 文件夹下是否存在 `user-profile.md` 和 `dish-library.md`。
- 两个文件都存在 → 跳到 **Step 2 的"非首次"分支**。
- 任一文件缺失 → 进入 **Step 2 的"首次"分支**。

### Step 2-A: 首次使用 — 引导建档

> ⚠️ 这一步**必须**通过提问获取信息，**不要替用户假设**。如果可用，使用 AskQuestion 工具一次性收集结构化答案；否则按以下顺序对话式提问。

请按以下顺序询问用户（可分组提问，不要一次塞太多）：

**A. 基础信息**
1. 用户的称呼/昵称？
2. 所在地区或时区（用于排周次）？
3. 是否有任何**忌口、过敏或宗教/伦理饮食限制**（如不吃牛羊肉、海鲜过敏、纯素等）？
4. 是否有需要长期遵循的**健康目标**（减肥 / 增肌 / 控糖 / 低钠 / 备孕 等）？

**B. 早餐结构**
5. 早餐由哪几类组成？（默认建议：肉类 + 蛋类 + 水果 + 主食，可增减）
6. 每类各上几样、份量大小？例如"蛋类 1 份、主食 1 份、水果 1 小碗"。
7. 早餐的烹饪时间预算（分钟）？

**C. 正餐结构**（用于午餐和晚餐）
8. 正餐由哪几类组成？（默认建议：肉菜 + 素菜，可加汤/主食）
9. 每餐通常上几个肉菜、几个素菜？是否需要主食/汤？
10. 午餐和晚餐结构是否一致？若不一致，请分别描述。

**D. 偏好与厨房现状**
11. 喜欢的食材清单（蛋白质、蔬菜、水果、主食各列几样）。
12. 不喜欢/不想常吃的食材。
13. 喜欢的菜系或口味（川/粤/淮扬/家常/低油少盐/西式…）。
14. 厨房工具（炒锅、烤箱、空气炸锅、电饭煲、破壁机…），用于过滤不可行做法。
15. 一周通常做几顿饭？哪几餐外食/外卖（这些餐次不规划）？
16. 单餐预计就餐人数。

**建档：**

收集完成后：
1. 读取 `templates/user-profile.template.md`，根据回答**填好后保存为** `user-profile.md`。
2. 读取 `templates/dish-library.template.md`，**至少为以下每个分类各预填 3-5 道符合用户偏好的菜品**，保存为 `dish-library.md`：
   - 早餐：肉类、蛋类、水果、主食
   - 正餐：肉菜、素菜
   - （若用户选择了汤/主食等额外分类，也建立对应分组）
3. 把生成的菜品库**展示给用户确认/补充/删减**。明确告知："菜品库会随使用持续迭代，你可以随时让我加菜、删菜或调整。"
4. 完成确认后，再进入 Step 3。

### Step 2-B: 非首次 — 加载档案

- 读取 `user-profile.md` 与 `dish-library.md`。
- 简要回顾："已加载你的偏好档案和 N 道菜的菜品库。本周有什么饮食期望？"

### Step 3: 解析"本周饮食期望"

把用户描述拆成结构化约束，例如：

| 期望类型 | 示例输入 | 转化为约束 |
|---------|---------|----------|
| 食材频率 | "这周想多吃鸡胸肉" | 至少出现 4 次 |
| 健康目标 | "在减肥" | 控制每餐热量、减少油炸/高糖、增加蔬菜与高蛋白 |
| 烹饪时间 | "工作日没空" | 工作日早餐 ≤ 10 分钟，晚餐 ≤ 30 分钟 |
| 重复度 | "别天天重样" | 同一道菜一周不超过 2 次 |
| 特定餐次 | "周日想吃顿好的" | 周日晚餐安排"招牌菜"分类 |

如果用户没说期望，按"日常均衡"模式：每天蛋白/蔬果/碳水均衡，单菜一周不超过 2 次。

### Step 4: 检查菜品库是否够用

对照本周约束扫一遍 `dish-library.md`：

- **够用** → 进入 Step 5。
- **不够用**（例如要求高频鸡胸肉但库中只有 1 道鸡胸菜）→ 提问："为了满足'多吃鸡胸肉'的目标，菜品库里只有 X 道鸡胸菜，你希望我加几道新菜吗？"用户同意后**先把新菜写入 `dish-library.md`，再回到 Step 5**。

> 严禁在最终食谱里出现菜品库中不存在的菜。

### Step 5: 生成一周食谱并写入"月份文件"

> 重要：**饮食计划按月份分文件**，每月只有一份 `meal-plan-YYYY-MM.md`，文件内通过"时间段（YYYY-MM-DD ~ YYYY-MM-DD）"小节区分各周。

#### 5.1 确定本周归属哪个月份文件

1. 算出本次要规划的"目标周"的**周一日期**（用户没指定就用"当前日期所在自然周"，周一为一周开始）。
2. 取这个**周一**的年月作为文件归属：文件名固定为 `meal-plan-YYYY-MM.md`（例如周一是 2026-04-27 → `meal-plan-2026-04.md`）。
3. 跨月规则：**只看周一**。即使周二到周日落在下个月，也归在周一所在月。
   - 示例 A：周一 2026-04-27、周日 2026-05-03 → 写入 `meal-plan-2026-04.md`。
   - 示例 B：周一 2026-05-04、周日 2026-05-10 → 写入 `meal-plan-2026-05.md`。
   - 示例 C：周一 2026-03-30、周日 2026-04-05 → 写入 `meal-plan-2026-03.md`。

#### 5.2 写入或追加

1. 读取 `templates/meal-plan.template.md`。
2. 检查 `meal-plan-YYYY-MM.md`：
   - **不存在** → 用模板新建该月文件，填充"月份信息（年月、归属规则说明）"，再追加本周小节。
   - **已存在** → **不要覆盖**。打开后追加一个新的周小节。
3. 周小节的固定标题格式：`## 周食谱 {{start_date}} ~ {{end_date}}`（例如 `## 周食谱 2026-04-27 ~ 2026-05-03`）。
4. 周小节内按用户档案中的"早餐结构"和"正餐结构"逐天填充菜品（仅从 `dish-library.md` 选）。
5. 周小节顶部写明：**本周期望、约束、生成日期**。每道菜后用 `[来源: dish-library.md#菜名]` 标注。
6. 周小节末尾追加**本周采购清单**（按"蛋白质 / 蔬菜 / 水果 / 主食 / 调味/其他"分组，合并同食材数量）。
7. 文件**末尾**维护一个"本月所有周"目录索引，新增周后追加一行链接锚点。
8. 默认把文件保存到 skill 文件夹下；**仅当用户在请求中明确指定其他路径**，才存到别处。

#### 5.3 同周重复规划的处理

- 若该月文件中已存在同一周（标题日期段完全相同）的小节：默认**就地替换**那一节内容（新生成覆盖旧的），并在小节末尾的"反馈记录"区追加 `重新生成于 {{datetime}}`。
- 若用户明确说"再排一版做对比"，则在原小节下追加 `### 备选方案 v2` 子节，不覆盖原方案。

### Step 6: 输出与迭代

- 在聊天中输出本周食谱**摘要**（每天早/午/晚一行）+ 完整文件路径，**不要把整个 markdown 表格贴回聊天**（避免冗余）。
- 主动询问 3 件事：
  1. 是否需要替换某天的某餐？
  2. 是否要把某些菜从库里移除（不喜欢/吃腻了）？
  3. 是否要新增几道菜进库？
- 用户给出反馈时：**直接修改 `dish-library.md` 和对应月份的 `meal-plan-YYYY-MM.md` 中的目标周小节**，不要新建文件。

## 模板使用说明

三个模板位于 `templates/`，由 agent 在第一次使用或新增条目时**读取**并按结构填写。模板里的 `{{占位符}}` 必须被替换为真实内容；不要把占位符留在最终文件里。

- `templates/user-profile.template.md` — 用户档案结构。
- `templates/dish-library.template.md` — 菜品库结构（含每道菜的字段：食材、热量估算、烹饪时间、做法要点、标签）。
- `templates/meal-plan.template.md` — 月份食谱文件结构：含月份头 + 一个或多个"周小节"（周表 + 详情 + 采购清单）+ 月底周索引。

## 迭代菜品库的规则

每次新增菜品到 `dish-library.md` 必须包含以下字段（与模板对齐）：

```
- 名称：
  分类：早餐-肉类 / 早餐-蛋类 / 早餐-水果 / 早餐-主食 / 正餐-肉菜 / 正餐-素菜 / 其他
  主要食材：
  烹饪时间(分钟)：
  估算热量(kcal/份)：
  口味标签：[低油][低盐][高蛋白][减脂友好][快手] 等
  做法要点：1-3 句话
  添加日期：YYYY-MM-DD
  用户评价：⭐ ⭐ ⭐ ⭐ ⭐（首次添加留空，吃过后由用户反馈打分）
```

- 用户说"这道菜不好吃 / 吃腻了" → 在条目末尾追加 `状态：已停用 (原因)`，**不要直接删除**，方便日后回看。
- 用户说"这道菜很喜欢" → 提升评价并加 `[最爱]` 标签，规划时优先级 +1。

---

## Anti-Patterns（不要这样做）

- ❌ 直接生成菜名却没有写入 / 不在 `dish-library.md` 里。
- ❌ 首次使用就猜测用户偏好建档（必须问）。
- ❌ 把整周食谱表格全贴回聊天造成刷屏。
- ❌ 因为某种食材"听起来健康"就加入计划（要看用户档案里是否喜欢、是否过敏）。
- ❌ 把用户档案、菜品库或食谱写到 skill 文件夹外（除非用户明确指定）。
- ❌ 为每周新建一个文件（应当按月份合并到 `meal-plan-YYYY-MM.md`，月内追加周小节）。
- ❌ 跨月时按"周日所在月"归档（必须看周一）。
- ❌ 写入新一周时覆盖掉同月已有的旧周小节（除非是同一周的重新规划）。

## 快速心智模型

> "我是一个**只会从已知食材柜里做菜的厨师助理**。第一次见面我会问清楚你的喜好，记到笔记本上；之后每次你告诉我这周想怎么吃，我就翻笔记和菜单，给你排出一份能直接照做的菜表和买菜清单。如果柜子里东西不够，我会先问你能不能补货，再开始排。"
