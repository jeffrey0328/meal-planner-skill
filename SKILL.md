# Meal Planner Skill (一周食谱规划)

本 skill 根据用户的**饮食偏好**和**当周饮食期望**，从一个**已建立的菜品库**中挑选菜品，生成定制化的一周食谱。

## 何时使用本 skill（重要）

**只在用户明确点名调用时才执行本 skill 的工作流。** 触发示例：

- "用 meal-planner-skill 帮我排食谱"
- "调用一下食谱规划 skill"
- "run meal-planner-skill"
- 用户在对话中明确提到 `meal-planner-skill` 这个名字

**不要**因为以下情形自动启动本 skill：

- 用户随口问"今晚吃什么"、"鸡胸肉怎么做"
- 用户聊到减肥、营养、菜谱话题但**没**点名本 skill
- 用户分享食谱链接、问烹饪技巧、查热量

如果不确定用户是否要调用本 skill，**先用一句话问清楚**："你是想让我用 meal-planner-skill 给你排一份正式的一周食谱（会读取/更新你的菜品库和档案），还是只是聊聊吃的？"

> 关键原则：**所有菜品必须来自**当前已解析的**菜品库文件**（见下文"数据路径"），不得凭空猜想。如果库中缺少能满足当周期望的菜品，必须先询问用户并把新菜加入库中，再规划。

## 数据路径（必读本节）

`user-profile.md` **始终、只能** 保存在 **skill 所在目录** 下。用户若指定「用户数据目录」，**仅**影响下面两类文件的位置；**其他任何东西都不得**写入用户目录。

| 可放在用户指定目录？ | 文件 | 说明 |
|---------------------|------|------|
| **是**（仅这二类） | `dish-library.md` | 菜品库 |
| **是**（仅这二类） | `meal-plan-YYYY-MM.md` | 按月的饮食计划 |
| **否** | `user-profile.md`、`dish-details.md`、模板、说明、本 SKILL、一切其他文件 | 一律只存在于 skill 目录；**不要**为了「方便」把档案/详情库写进用户目录 |

**如何知道用户数据目录在何处：** 打开 skill 下的 `user-profile.md`，看 **「0. 数据文件位置」** 小节。若 `菜品库与饮食计划根目录` 为**空**或写明「与 skill 同目录」，则上述两类文件与 `user-profile` 同目录。若填了**绝对路径**（如 `D:\MyMealData`），则仅在该路径下读写 `dish-library.md` 与 `meal-plan-*.md`。

**在对话中由用户随时指定「换目录」** 时：只更新 `user-profile.md` 里该路径字段，之后所有读写都跟随新路径；**不得**在旧路径或新路径里**新建** `user-profile.md` 或**除上述二类以外的文件**。

**若用户数据目录尚不存在**：可创建该**空目录**以便写入，但**不得**在其中创建除 `dish-library.md` / `meal-plan-*.md` 外的任何文件（不建 `.keep`、不建 README、不建日志等）。

在食谱/引用里写来源时，用**实际路径**，例如：`` `D:\MyMealData\dish-library.md#菜名` `` 或相对 skill 时的约定一致表述。

## 文件结构

skill 文件夹下**固定**会有（未存在的会按需创建）：

```
meal-planner-skill/
├── CHANGELOG.md
├── SKILL.md
├── templates/
│   ├── user-profile.template.md
│   ├── dish-library.template.md
│   ├── dish-details.template.md
│   └── meal-plan.template.md
└── user-profile.md                # 仅此处；可记录「用户数据目录」
```

`CHANGELOG.md` 用于记录**skill 定义**的更新（按天分节）。凡是修改了 `SKILL.md` / 模板 / README 等定义文件，都应在当天追加 1 条更新日志；用户数据的日常内容更新不写入更新日志。

**下面两类文件**要么与 `user-profile.md` 同目录，要么仅在用户配置的「数据根目录」下（**仅此二类**）：

- `dish-library.md`
- `meal-plan-YYYY-MM.md`（按月份，月内按周；跨月周以**周一**所在月为准，见 Step 5）

`dish-details.md`（菜品详情库）**只**在 `SKILL_DIR` 下存在，用于提供热量/食材/评价等信息；`dish-library.md` 只保存菜名。

## 主工作流（每次被调用都按此执行）

复制以下检查清单并跟踪进度：

```
任务进度：
- [ ] Step 0: 从 user-profile 解析「数据根目录」
- [ ] Step 1: 判断是否首次使用
- [ ] Step 2: 首次 → 引导建档；非首次 → 加载档案
- [ ] Step 3: 解析本次"周饮食期望"
- [ ] Step 4: 检查菜品库是否够用，必要时补充
- [ ] Step 5: 生成一周食谱并写入月份文件
- [ ] Step 6: 输出概要并询问是否需要迭代
```

### Step 0: 解析数据根目录

1. 令 `SKILL_DIR` = 本 skill 根目录（含 `SKILL.md` 的目录）。
2. 若存在 `SKILL_DIR/user-profile.md`：读取其 **「0. 数据文件位置」** 中 **「菜品库与饮食计划根目录」**。
3. 若该字段**为空**、或写「与 skill 同目录」/「默认」→ **`DATA_ROOT` = `SKILL_DIR`**。
4. 若该字段为**非空绝对路径** → **`DATA_ROOT` = 该路径**（规范化后，不要尾随 `\` 歧义）。
5. 后续步骤中：
   - **菜品库文件** = `DATA_ROOT/dish-library.md`（用平台路径分隔符；Windows 上写 `\` 亦可）。
   - **月份饮食计划** = `DATA_ROOT/meal-plan-YYYY-MM.md`。
6. **菜品详情库** = `SKILL_DIR/dish-details.md`（永不写入 `DATA_ROOT`）。
7. **`user-profile.md` 的路径永远是 `SKILL_DIR/user-profile.md`**，不随 `DATA_ROOT` 变化。

### Step 1: 判断是否首次使用

- 检查 **`SKILL_DIR/user-profile.md`** 是否存在，以及 **`DATA_ROOT/dish-library.md`** 是否存在，以及 **`SKILL_DIR/dish-details.md`** 是否存在。  
  （若 `user-profile` 尚未存在，`DATA_ROOT` 在首次建档前可视为 `SKILL_DIR`，直到用户明确选了别的目录并写入 `user-profile` 第 0 节。首次建档的提问顺序里**先**问清是否用自定义 `DATA_ROOT`，**再**在对应位置创建 `dish-library.md`。）
- `user-profile` 与（解析后的）`dish-library` 都存在 → **非首次**。
- 至少缺其一 → **首次**（进入 Step 2-A 引导建档，建档过程中补齐路径与第 0 节后再建库）。

### Step 2-A: 首次使用 — 引导建档

> ⚠️ 这一步**必须**通过提问获取信息，**不要替用户假设**。如果可用，使用 AskQuestion 工具一次性收集结构化答案；否则按以下顺序对话式提问。

请按以下顺序询问用户（可分组提问，不要一次塞太多）：

**A. 基础信息**
1. 用户的称呼/昵称？
2. 所在地区或时区（用于排周次）？
3. 是否有任何**忌口、过敏或宗教/伦理饮食限制**（如不吃牛羊肉、海鲜过敏、纯素等）？
4. 是否有需要长期遵循的**健康目标**（减肥 / 增肌 / 控糖 / 低钠 / 备孕 等）？

**B. 早餐结构**
5. 早餐由哪几类组成？（默认建议：肉类 + 蛋类 + 水果 + 主食，可增减）
6. 每类各上几样、份量大小？例如"蛋类 1 份、主食 1 份、水果 1 小碗"。
7. 早餐的烹饪时间预算（分钟）？

**C. 正餐结构**（用于午餐和晚餐）
8. 正餐由哪几类组成？（默认建议：肉菜 + 素菜，可加汤/主食）
9. 每餐通常上几个肉菜、几个素菜？是否需要主食/汤？
10. 午餐和晚餐结构是否一致？若不一致，请分别描述。

**D. 偏好与厨房现状**
11. 喜欢的食材清单（蛋白质、蔬菜、水果、主食各列几样）。
12. 不喜欢/不想常吃的食材。
13. 喜欢的菜系或口味（川/粤/淮扬/家常/低油少盐/西式…）。
14. 厨房工具（炒锅、烤箱、空气炸锅、电饭煲、破壁机…），用于过滤不可行做法。
15. 一周通常做几顿饭？哪几餐外食/外卖（这些餐次不规划）？
16. 单餐预计就餐人数。

**E. 数据文件位置**（在其它问题都清楚后再问，避免与路径混淆）
17. 是否将 **`dish-library.md`** 和 **`meal-plan-*.md`** 存放到**你指定的根目录**？（`user-profile.md` **不**会放在该目录，仍只在 skill 目录。）  
   - 若**否** → 在 `user-profile` 第 0 节写「与 skill 同目录 / 默认」。  
   - 若**是** → 请用户提供**希望使用的绝对路径**；说明**仅**会在该目录读写的文件是：菜品库 + 月饮食计划，**不会**把档案、模板或其它任何文件放过去。
18. 若用户给的路径不存在：可创建**空目录**以便写入，但**不要**往该目录里放除 `dish-library.md` / `meal-plan-*.md` 外的任何内容。

**建档：**

收集完成后：
1. 读取 `templates/user-profile.template.md`，**填写第 0 节（数据文件位置）**与全文，**保存为** `SKILL_DIR/user-profile.md`（**禁止**在 `DATA_ROOT` 下保存或复制本档案）。
2. 根据第 0 节解析出 `DATA_ROOT`（同 Step 0 规则），读取 `templates/dish-library.template.md`，**按分类预填菜名**，保存为 **`DATA_ROOT/dish-library.md`**（本文件只含菜名）。
3. 读取 `templates/dish-details.template.md`，为上述菜名逐条补齐详情（热量/食材/添加日期/评价/状态），保存为 **`SKILL_DIR/dish-details.md`**（仅此处；不要写到 `DATA_ROOT`）。
   - 早餐：肉类、蛋类、水果、主食
   - 正餐：肉菜、素菜
   - （若用户选择了汤/主食等额外分类，也建立对应分组）
4. 把生成的菜名库**展示给用户确认/补充/删减**；并说明：详情在 `dish-details.md`，后续加菜会同时更新两处。
5. 完成确认后，再进入 Step 3。

> 非首次、用户**以后**在对话中要求**更换** `DATA_ROOT` 时：只改 `user-profile.md` 第 0 节；**不要**在旧目录**新建** `user-profile` 或任何杂项。若新目录尚未有 `dish-library.md`，**询问**是迁移旧库还是新建（用户确认后再操作）；迁移时只复制/移动**允许的**两类文件，不复制 `user-profile` 到数据目录。

### Step 2-B: 非首次 — 加载档案

1. 先按 **Step 0** 重算 `DATA_ROOT`（已读入 `user-profile` 后路径即确定）。  
2. 读取 `SKILL_DIR/user-profile.md`、`DATA_ROOT/dish-library.md` 与 `SKILL_DIR/dish-details.md`。  
3. 简要回顾："已加载你的偏好档案和 N 道菜的菜品库（库路径：…）。本周有什么饮食期望？"

### Step 3: 解析"本周饮食期望"

把用户描述拆成结构化约束，例如：

| 期望类型 | 示例输入 | 转化为约束 |
|---------|---------|----------|
| 食材频率 | "这周想多吃鸡胸肉" | 至少出现 4 次 |
| 健康目标 | "在减肥" | 控制每餐热量、减少油炸/高糖、增加蔬菜与高蛋白 |
| 烹饪时间 | "工作日没空" | 工作日早餐 ≤ 10 分钟，晚餐 ≤ 30 分钟 |
| 重复度 | "别天天重样" | 同一道菜一周不超过 2 次 |
| 特定餐次 | "周日想吃顿好的" | 周日晚餐安排"招牌菜"分类 |

如果用户没说期望，按"日常均衡"模式：每天蛋白/蔬果/碳水均衡，单菜一周不超过 2 次。

### Step 4: 检查菜品库是否够用

对照本周约束扫一遍 **`DATA_ROOT/dish-library.md`**（Step 0 已解析）：

- **够用** → 进入 Step 5。
- **不够用**（例如要求高频鸡胸肉但库中只有 1 道鸡胸菜）→ 提问："为了满足'多吃鸡胸肉'的目标，菜品库里只有 X 道鸡胸菜，你希望我加几道新菜吗？"用户同意后**先把新菜写入**上述路径的菜品库文件，**再回到 Step 5**。

> 严禁在最终食谱里出现菜品库中不存在的菜。

### Step 5: 生成一周食谱并写入"月份文件"

> 重要：**饮食计划按月份分文件**，每月只有一份 `meal-plan-YYYY-MM.md`，文件内通过"时间段（YYYY-MM-DD ~ YYYY-MM-DD）"小节区分各周。

#### 5.1 确定本周归属哪个月份文件

1. 算出本次要规划的"目标周"的**周一日期**（用户没指定就用"当前日期所在自然周"，周一为一周开始）。
2. 取这个**周一**的年月作为文件归属：文件名固定为 `meal-plan-YYYY-MM.md`（例如周一是 2026-04-27 → `meal-plan-2026-04.md`）。
3. 跨月规则：**只看周一**。即使周二到周日落在下个月，也归在周一所在月。
   - 示例 A：周一 2026-04-27、周日 2026-05-03 → 写入 `meal-plan-2026-04.md`。
   - 示例 B：周一 2026-05-04、周日 2026-05-10 → 写入 `meal-plan-2026-05.md`。
   - 示例 C：周一 2026-03-30、周日 2026-04-05 → 写入 `meal-plan-2026-03.md`。

#### 5.2 写入或追加

1. 读取 `templates/meal-plan.template.md`。
2. 月份文件路径 = **`DATA_ROOT/meal-plan-YYYY-MM.md`**（`DATA_ROOT` 由 Step 0 确定；与 `dish-library.md` 始终同一根目录）。  
3. 检查该路径是否已存在 `meal-plan-YYYY-MM.md`：
   - **不存在** → 用模板新建该月文件，填充"月份信息（年月、归属规则说明，可注明 `数据根目录: DATA_ROOT`）"，再追加本周小节。
   - **已存在** → **不要整文件覆盖**。打开后**追加**一个新的周小节。
4. 周小节的固定标题格式：`## 周食谱 {{start_date}} ~ {{end_date}}`（例如 `## 周食谱 2026-04-27 ~ 2026-05-03`）。
5. 周小节内按用户档案中的"早餐结构"和"正餐结构"逐天填充菜品：\n   - **选菜来源**：只从 `DATA_ROOT/dish-library.md` 里出现过的菜名中选。\n   - **详情来源**：从 `SKILL_DIR/dish-details.md` 查对应条目（热量/食材/评价/状态）。若菜名在库里但详情缺失，先问用户补齐并写入详情库，再生成食谱。\n   - **午餐/晚餐展示规则**：若用户档案明确写了“午餐和晚餐一致/相同”，则在输出文件中**不重复写两遍**，统一使用“正餐”一栏（周表为：日期/早餐/正餐；每日详情为：早餐 + 正餐）。只有当用户档案明确表示午餐与晚餐不同，才拆分为“午餐/晚餐”两栏与两段详情。\n+6. 周小节顶部写明：**本周期望、约束、生成日期**。每道菜后标注来源，须含**可定位路径**，例如：`` [来源: D:\Data\dish-library.md#菜名; 详情: SKILL_DIR\\dish-details.md#菜名] ``。
7. 周小节末尾追加**本周采购清单**（按"蛋白质 / 蔬菜 / 水果 / 主食 / 调味/其他"分组，合并同食材数量）。
8. 文件**末尾**维护一个"本月所有周"目录索引，新增周后追加一行链接锚点。

> **禁止**把 `meal-plan-*.md` 写到 `DATA_ROOT` 以外；**禁止**因「想单独指定食谱路径」在别处再建第二套根目录，除非用户**先**通过修改 `user-profile` 第 0 节更换 `DATA_ROOT`（与菜品库同根，一体变更）。

#### 5.3 同周重复规划的处理

- 若该月文件中已存在同一周（标题日期段完全相同）的小节：默认**就地替换**那一节内容（新生成覆盖旧的），并在小节末尾的"反馈记录"区追加 `重新生成于 {{datetime}}`。
- 若用户明确说"再排一版做对比"，则在原小节下追加 `### 备选方案 v2` 子节，不覆盖原方案。

### Step 6: 输出与迭代

- 在聊天中输出本周食谱**摘要**（每天早/午/晚一行）+ 完整文件路径，**不要把整个 markdown 表格贴回聊天**（避免冗余）。
- 主动询问 3 件事：
  1. 是否需要替换某天的某餐？
  2. 是否要把某些菜从库里移除（不喜欢/吃腻了）？
  3. 是否要新增几道菜进库？
- 用户给出反馈时：**直接修改** `DATA_ROOT/dish-library.md` 与同目录下对应月份 **`DATA_ROOT/meal-plan-YYYY-MM.md`** 中的目标周小节，**不要**新建**未经允许的**文件（对「用户目录」只许改这两类；`user-profile` 仍在 `SKILL_DIR`）。

---

## 模板使用说明

三个模板位于 `templates/`，由 agent 在第一次使用或新增条目时**读取**并按结构填写。模板里的 `{{占位符}}` 必须被替换为真实内容；不要把占位符留在最终文件里。

- `templates/user-profile.template.md` — 用户档案结构。
- `templates/dish-library.template.md` — 菜名库：只包含各分类的「菜名速查区（仅菜名，多列一行便于快速查阅；无标题、无表头文字）」。不含详情字段。
- `templates/dish-details.template.md` — 菜品详情库：按分类维护每道菜的详情（热量、所需食材、添加日期、用户评价、状态/停用原因）。
- `templates/meal-plan.template.md` — 月份食谱文件结构：含月份头 + 一个或多个"周小节"（周表 + 详情 + 采购清单）+ 月底周索引。

## 迭代菜品库的规则

每次新增菜品时：\n- 先把菜名加入 `DATA_ROOT/dish-library.md` 对应分类的速查区\n- 再把详情写入 `SKILL_DIR/dish-details.md`（与模板对齐）。\n\n每个详情条目必须包含以下最小字段：

```
- 名称：
  分类：早餐-肉类 / 早餐-蛋类 / 早餐-水果 / 早餐-主食 / 正餐-肉菜 / 正餐-素菜 / 其他
  热量(kcal/份)：
  所需食材：
  添加日期：YYYY-MM-DD
  用户评价：⭐ ⭐ ⭐ ⭐ ⭐（首次添加可留空，吃过后由用户反馈打分）
  状态：启用 / 已停用 (原因)
```

- 用户说"这道菜不好吃 / 吃腻了" → 在详情里把 `状态` 改为 `已停用 (原因)`；**不要直接删除**，方便日后回看。（菜名表格仍保留该菜名，便于历史查阅）
- 用户说"这道菜很喜欢" → 提升详情里的 `用户评价`（⭐）。

## Anti-Patterns（不要这样做）

- ❌ 直接生成菜名却没有写入 / 不在当前 `DATA_ROOT/dish-library.md` 里。
- ❌ 首次使用就猜测用户偏好建档（必须问）。
- ❌ 把整周食谱表格全贴回聊天造成刷屏。
- ❌ 因为某种食材"听起来健康"就加入计划（要看用户档案里是否喜欢、是否过敏）。
- ❌ 在**用户为数据根目录**里创建 **`user-profile.md`、说明文档、配置副本、.keep、日志** 或**除** `dish-library.md` / `meal-plan-*.md` **以外的任何文件**。
- ❌ 把 `user-profile.md` 存到 `DATA_ROOT` 或**任何**与 skill 分离的仅「用户说方便」的位置（`user-profile` 只能在 `SKILL_DIR`）。
- ❌ 把**菜品库或饮食计划**存到**非** `user-profile` 第 0 节所约定根目录的地方（不拆散到多个自定义路径）。
- ❌ 为每周新建一个文件（应当按月份合并到 `meal-plan-YYYY-MM.md`，月内追加周小节）。
- ❌ 跨月时按"周日所在月"归档（必须看周一）。
- ❌ 写入新一周时覆盖掉同月已有的旧周小节（除非是同一周的重新规划）。
